# “校缘”高校校友相亲平台后台管理系统产品需求文档

## 1. 系统愿景与定位

“校缘”后台管理系统是专为运营团队与各试点高校联络人设计的管理工具。其定位为“**严肃相亲生态的守护者与校友活动运营中枢**”。
通过严格的身份审核、闭环的举报处理，确保平台数据的真实与纯净；通过分校管理和细粒度的数据隔离，支持多校联合运营；通过灵活的活动及 Banner 投放，促进单身校友的同频互动与社群活跃。

---

## 2. 角色与数据隔离权限模型

### 2.1 角色定义
系统采用基于角色的访问控制（RBAC）模型，一期 MVP 仅划分以下两个管理角色：
1. **超级管理员 (super_admin)**：平台拥有者与总运营。负责全局数据监控、系统级配置及跨校事务处理。
2. **学校管理员 (school_admin)**：特定高校的校友会负责人或联络人（如“南京大学联络人”）。负责本校校友的日常资料审核、违规处理及专属活动发布。

### 2.2 数据隔离矩阵 (Data Isolation Matrix)

为了确保各试点高校数据的相对独立性与校友隐私，系统实施严格的数据隔离机制：

| 功能模块 | 数据维度 | 超级管理员 (super_admin) | 学校管理员 (school_admin) |
| --- | --- | --- | --- |
| **控制台大盘** | 用户与活跃统计 | 展现全平台各校汇总及对比数据。 | **仅展示所属学校**（`admin_school`）的统计指标。 |
| **用户管理** | 资料查看与管理 | 查看、搜索、冻结全平台任意用户。 | **仅可查看及冻结所属学校**（`school = admin_school`）的用户。 |
| **认证审核** | 资料核验与审批 | 可审批全平台所有用户的四项审核申请。 | **仅能审批所属学校**（`school = admin_school`）用户的申请。 |
| **活动管理** | 线下活动发布及报名 | 管理全平台所有活动，导出全量报名表。 | **仅能发布及管理本校专属活动**，导出本校活动的报名表。 |
| **Banner配置** | 首页轮播图配置 | 拥有全局 Banner 增删改查权限。 | **无权访问该模块**（不可查看及修改）。 |
| **心动记录** | 匹配日志监控 | 监控全平台所有心动申请及问答内容。 | 仅展示**发起方或接收方至少有一方为本校用户**的记录。 |
| **消息中心监控** | 消息记录（点赞、心动通知） | 监控全平台所有点赞及心动通知流。 | 仅展示**收发双方至少有一方为本校用户**的通知记录。 |
| **举报与违规** | 投诉受理与封禁 | 受理全平台所有举报，判定封禁。 | **仅能处理被举报人为本校用户**的举报。 |
| **操作日志** | 审计日志记录 | 查看全平台所有管理员的操作日志。 | 仅能查看本账号的操作历史。 |

> [!IMPORTANT]
> **隔离技术实现规约**：
> - 接口层必须强制拦截：学校管理员发起的所有查询和修改请求，后端云函数必须自动在数据库查询条件（Query）中注入 `school = admin_school` 过滤条件，不得依赖前端传递的学校参数，从根本上防止越权访问漏洞（IDOR）。

## 3. 核心功能模块明细

### 3.1 控制台与数据大盘 (Dashboard)
为管理员提供直观的指标概览，监控平台日常运行与活跃。
* **核心指标卡片**：
  * **用户总数**（及未认证用户占比）。
  * **实名率与学历认证率**。
  * **心动匹配率**：心动申请总数，其中 `accepted`（接受）申请的比例。
  * **日活（DAU）/ 月活（MAU）**。
* **统计图表**：
  * **用户增长曲线**（支持按日、周、月筛选）。
  * **活跃趋势折线图**。
  * *（仅超级管理员可见）*：各试点学校的用户分布饼图与匹配率对比柱状图。

### 3.2 用户管理模块 (User Management)
提供对单身狗和介绍人账号的日常管理与状态维护。
* **搜索与筛选条件**：
  * 支持输入姓名、昵称、手机号、微信号进行精准搜索。
  * 支持按性别、认证状态（实名已核、学历已核、未认证）、账号状态（正常、已冻结）进行筛选。
  * *（超级管理员独占）*：支持按学校筛选。学校管理员默认锁定本校。
* **用户资料详页**：
  * 集中展示用户的实名信息、学历信息、工作与婚姻状况。
  * **敏感数据脱敏展示**：身份证号明文不展示，仅展示出生年份与前4位后4位打码值（如 `3201**********0012`）。
  * 展示用户的 3 个心动问题、自我介绍及个人相册。
* **管理操作**：
  * **冻结账号**：限制违规用户登录，点击弹出冻结原因输入框（冻结原因将同步至用户小程序端）。
  * **解冻账号**：恢复用户状态。

### 3.3 用户审核中心 (Verification Review)
本系统的一期 MVP 核心工作流，用于处理用户提交的各项真实性证明。

```mermaid
flowchart TD
    A[用户提交审核申请] --> B{管理员角色?}
    B -->|超级管理员| C[可审核全部申请]
    B -->|学校管理员| D{申请人属于本校?}
    D -->|是| E[可审核该申请]
    D -->|否| F[列表无数据显示/无权操作]
    C & E --> G[审核操作]
    G -->|核验无误| H[点击"通过"]
    G -->|资料有误| I[点击"驳回"]
    I --> J[选择/输入驳回具体原因]
    H --> K[更新Profile对应认证状态为已认证并解锁锁定字段]
    J --> L[用户小程序端收到驳回提示与重新编辑引导]
```

* **审核列表**：
  * 按类型分为：**实名认证审核**（对于人工复核备用流）、**学历认证审核**、**工作认证审核**。
  * 列表项展示：申请人、提交时间、证明材料附件（学生证/毕业证/工牌等照片）。
* **审核操作**：
  * **审核通过**：点击确认通过，更新状态，自动为用户点亮对应认证标签（如“已实名”绿标、“东大认证”/“高校已核”蓝标）。
  * **审核驳回**：点击弹出驳回弹窗。管理员必须输入或选择具体的**驳回原因**（例如：“证明文件模糊不清，请重新拍照上传”、“工牌名字与实名不符”）。驳回原因将通过小程序通知反馈给用户，并引导用户重新编辑提交。

### 3.4 活动与 Banner 配置管理 (Activity & Banner)
* **活动管理（分校独立发布）**：
  * **活动列表**：展示活动的标题、发布时间、报名人数上限、当前已报名人数、活动状态（草稿、征集中、进行中、已结束）。
  * **发布/编辑活动**：
    * 字段：活动标题、活动主图、活动时间、地点、限制学校（超级管理员可选择“全部学校”或具体某校；学校管理员强制锁定本校）、报名人数上限、活动介绍（富文本/Markdown 格式）。
  * **报名名单管理**：
    * 点击某活动查看已报名人员列表。
    * 提供**“导出 Excel 报名表”**功能，便于线下活动现场签到与人员核对。
* **Banner 配置（超级管理员专属）**：
  * **Banner 列表**：展示当前 Banner 幻灯片（图片预览、跳转链接、排序权重、是否启用）。
  * **配置操作**：新建 Banner、上传图片（规格 750x300 像素）、设置跳转路由（如具体活动详情页或认证引导页）、启用/禁用开关、拖拽排序权重。

### 3.5 心动记录监控 (Heart Records)
用于评估平台匹配成效与分析用户互动的关键模块。
* **数据列表**：
  * 列表展示：心动发起方（A）、接收方（B）、发起时间、状态（处理中 `pending`、已接受 `accepted`、已拒绝 `rejected`）、处理时间。
  * 点击单条记录可展开**“破冰问答面板”**，查看 A 针对 B 的 3 个心动问题的具体回答文本。
* **数据隔离细节**：
  * 学校管理员只能看到 `A.school == admin_school` 或 `B.school == admin_school` 的记录，便于跟进和撮合本校同学。

### 3.6 违规举报与安全管理 (Security Reports)
保障高信任社区纯净度的关键风控防线。
* **举报列表**：
  * 举报条目：被举报人、举报发起人、举报原因分类（资料虚假、言行骚扰、广告引流、其他）、文字描述及上传截图证明、提交时间、处理状态（待处理、已处罚、已忽略）。
* **处理操作**：
  * **忽略**：判定举报不成立，标记已读。
  * **警告**：向被举报人下发系统警告提示。
  * **处罚（冻结）**：一键将该被举报人账号进行冻结封禁，使其相亲卡从瀑布流中淡出，防止继续危害社区。

### 3.7 系统审计日志 (Audit Logs)
* 记录管理员登录、审批通过/驳回、冻结用户、发布活动、修改配置等关键操作。
* 记录格式：`[操作时间] 管理员 [账号名] (学校名) 对 [目标对象] 进行了 [操作类型] 操作，详情：[日志文本]`。防止管理员恶意审批或违规篡改数据。

### 3.8 消息通知监控中心 (Message Logs)
用户端的所有积极互动（点赞与回答心动问题）均会在数据库持久化生成消息通知，管理后台提供只读的消息流水监控，用于评估平台活跃度。
* **数据展示列表**：
  - 字段：消息 ID、发送方、接收方、消息类型（点赞 / 心动）、关联心动申请 ID（仅心动消息）、发送时间、已读状态。
* **数据隔离规则**：
  - 学校管理员：仅能检索其辖区内学校用户相关的通知记录（发送方或接收方学校属于 `admin_school`）。
  - 超级管理员：可全局检索、按学校汇总统计日均点赞量、心动次数等关键活跃指标。

## 4. 后台页面拓扑结构与交互模型

管理后台页面采用经典的**左侧侧边栏导航 + 右侧内容区**的主流布局：

```text
管理后台/
├── 1. 控制台大盘 (pages/dashboard)
├── 2. 用户管理 (pages/users)
│   └── 详情卡片弹窗
├── 3. 审核中心 (pages/reviews)
│   ├── 实名人工复核
│   ├── 学历审核列表
│   └── 工作审核列表
├── 4. 运营管理 (pages/operations)
│   ├── 活动管理列表 --> 发布/编辑活动页面 --> 报名名单与导出
│   └── 首页Banner配置 (仅超级管理员可见)
├── 5. 匹配监控 (pages/match-logs)
│   └── 问答明细展开
├── 6. 消息通知监控 (pages/message-logs)
└── 7. 安全风控 (pages/security)
    └── 举报投诉受理列表
```

* **交互设计准则**：
  - 列表页面均需具备一键重置筛选、分页及每页条数选择。
  - 核心操作（如审核通过、驳回、冻结、删除活动）必须伴有**二次确认提示框**，防止手误。
  - 暖色调主旋律：UI 界面采用简洁清爽的淡色设计，按钮及高亮边框使用与小程序一致的珊瑚橙/暖蜜橙，强化产品系列统一性。

## 5. 数据模型扩展定义

管理后台的引入，要求在原有的小程序数据模型中扩展以下数据结构：

### 5.1 活动表 (Activity)
* `id`: Int (主键)
* `title`: String (活动标题)
* `cover_url`: String (活动封面图)
* `start_time`: DateTime (开始时间)
* `end_time`: DateTime (结束时间)
* `address`: String (活动地点)
* `limit_school`: String (限制可见的高校，'all' 或 '南京大学' / '江南大学')
* `max_participants`: Int (报名人数上限)
* `current_participants`: Int (当前报名人数)
* `content`: Text (活动详情介绍，支持 Markdown)
* `creator_id`: Int (发布者管理员ID)
* `status`: String (draft/active/ended)

### 5.2 首页 Banner 表 (Banner)
* `id`: Int (主键)
* `image_url`: String (Banner图片URL)
* `redirect_url`: String (点击跳转的小程序路由，如 `/pages/activity/detail?id=1`)
* `sort_weight`: Int (排序权重，数字越大越靠前)
* `status`: String (enabled/disabled)
* `creator_id`: Int (操作者ID)

### 5.3 举报投诉表 (Report)
* `id`: Int (主键)
* `reporter_id`: Int (举报发起人 ID)
* `reported_user_id`: Int (被举报人 ID)
* `reason_category`: String (举报类别)
* `description`: Text (文字描述)
* `screenshot_urls`: List~String~ (证明截图列表)
* `status`: String (pending/processed/ignored)
* `handle_result`: String (处罚结果：warned/frozen/none)
* `handler_id`: Int (处理的管理员 ID)
* `submit_time`: DateTime

### 5.4 操作日志表 (AuditLog)
* `id`: Int (主键)
* `admin_id`: Int (管理员 ID)
* `action_type`: String (操作类型，如 approve_edu / freeze_user / create_activity)
* `target_id`: String (操作的目标对象 ID)
* `details`: Text (操作详情日志描述)
* `ip_address`: String (操作时 IP)
* `log_time`: DateTime
